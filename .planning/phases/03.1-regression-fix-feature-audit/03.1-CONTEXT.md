# Phase 3.1: Regression Fix & Feature Audit - Context

**Gathered:** 2026-03-28 (auto mode)
**Status:** Ready for planning

<domain>
## Phase Boundary

Systematically verify and fix 18+ regressions discovered during user testing. Every feature claimed as "shipped" in prior phases must be verified at runtime on a fresh build. Fix all breakage, restore missing UI access points, correct labeling, and address specific UX issues on the Voice Confirm Screen and transaction forms.

**Key insight:** A code audit found all 18 features exist in source code — but the user physically tested the app and reported them broken/missing. This phase bridges the gap between "code exists" and "feature works for the user."

</domain>

<decisions>
## Implementation Decisions

### Verification Methodology
- **D-01:** Build a fresh debug APK and verify each feature on an emulator step-by-step. Code existing ≠ feature working. Every item gets a manual runtime test.
- **D-02:** Create a checklist-based verification plan. Each of the 18+ items gets: (a) verify at runtime, (b) identify the root cause if broken, (c) fix, (d) re-verify.
- **D-03:** If a feature works correctly and the user's report was due to a stale build, mark it as "verified — no fix needed" in the plan output. Do not skip verification.

### Label Naming: "Subscriptions & Bills" (Not "Recurring & Bills")
- **D-04:** Change ALL user-facing labels from "Recurring & Bills" to "Subscriptions & Bills". This includes: l10n keys (`recurring_and_bills_title` → `subscriptions_and_bills_title`), nav tabs, screen titles, and any UI text. User explicitly requested this rename.
- **D-05:** Internal code (class names, file names, DB table name `recurring_rules`) stays as-is. Only user-facing text changes. No file renames — that's unnecessary churn.

### Cash Wallet Visibility
- **D-06:** Verify the system Cash wallet appears in the home screen account chips. If the Phase 3 home screen overhaul excluded system wallets from the chip list, restore visibility. Cash must be a selectable filter option — users need to differentiate physical cash from bank accounts.
- **D-07:** Verify AI voice/chat correctly resolves "cash", "كاش", and Arabizi variants to the system Cash wallet.

### Account Management Features (Reorder, Archive, Default)
- **D-08:** Verify the "Manage" button (or equivalent entry point) that opens `account_manage_sheet.dart` is accessible from the new home screen. If the old carousel had a "Manage" button that was lost in the chip-based redesign, add an entry point (e.g., long-press on chip row, or a gear icon).
- **D-09:** Verify drag-and-drop reorder works at runtime inside the management sheet.
- **D-10:** Verify archive/unarchive works: archive an account → it disappears from chips → unarchive → it reappears. Verify archived accounts are excluded from AI context, analytics, and balance calculations.
- **D-11:** Verify "set as default" is accessible and functional. Default account should be visually distinguished.

### Home Screen & RTL
- **D-12:** Verify balance display and account chips are center-aligned (not left-aligned) in both LTR and RTL modes. If the user reports LTR alignment, this is a rendering issue — check the actual widget tree, not just the source for `Alignment` keywords.
- **D-13:** Test the full home screen in Arabic RTL on emulator. Verify: balance header, account chips, filter bar, transaction list, date headers, swipe actions all render correctly.

### Voice Confirm Screen UX Fixes
- **D-14:** Fix transfer layout: verify font sizes use design tokens (not hardcoded). If the user sees huge fonts and duplicate amounts, there's a rendering bug in the transfer card layout.
- **D-15:** Fix multi-account creation on transfer: when voice says "transfer from X to Y" and BOTH X and Y don't exist, suggest creating BOTH accounts — not just one. Show create-account suggestions inside the wallet dropdown for the most seamless flow (fewer steps).
- **D-16:** Add "Accept All" button for multi-draft review. When voice parses multiple transactions, user should be able to save all at once instead of one-by-one. This was explicitly requested: "less steps, more accessibility, ease of use."
- **D-17:** Category should be auto-assigned based on voice parsing. Show it pre-selected with option to change. Support selecting multiple categories if ambiguous (Claude's discretion on implementation).

### Transaction Form: "Additional Details" Toggle
- **D-18:** Evaluate the current "Optional Details" expandable section. The user wants this rethought as a toggle approach. Verify the current implementation at runtime — if it works well and is discoverable, it may just need better labeling or a toggle switch instead of an expand/collapse animation.

### SnackBar / Toast Position
- **D-19:** Verify SnackBars appear at the bottom of the screen, not mid-screen. Check `AppSizes.snackbarBottomMargin` value — it may be too large, pushing the snackbar to the center. Also verify the floating behavior doesn't conflict with the glassmorphic nav bar.

### AI Language Behavior
- **D-20:** Verify the Arabizi system prompt instruction works at runtime. Test: type "ana sraft 200 gneih" → AI should reply in Arabic.
- **D-21:** Verify AI replies in the same language the user types. English input → English reply. Arabic input → Arabic reply. Arabizi input → Arabic reply. This should work regardless of app locale setting.
- **D-22:** If the prompt-based approach doesn't work reliably with the current OpenRouter models (Gemma/Qwen), add a code-level language hint that detects Arabizi patterns (numbers-for-letters: 3=ع, 7=ح, 2=ء) and passes `respond_in: "ar"` to the system prompt.

### Brand Icons
- **D-23:** Verify brand icons render correctly at runtime. The brand registry uses keyword→initial mappings (colored circles with initials like "VC" for Vodafone Cash). Verify these display on transaction cards when a matching brand is detected.
- **D-24:** Actual logo assets (bank logos, app logos, merchant logos) are out of scope for this fix phase. The current initial-based approach is the shipped design. If the user wants actual logos, that's a separate feature request for a future phase.

### Subscriptions & Bills Screen
- **D-25:** Verify the screen renders correctly with all sections (Overdue, Upcoming Bills, Active, Paid, Paused). If the user says "you haven't changed anything" — the screen may be correct as-is from prior phases, or it may need UI polish that was promised but not delivered. Verify against Phase 3/4 CONTEXT.md promises.

### Transaction Details & CRUD Screens
- **D-26:** Verify transaction detail screen shows correct data, edit/delete actions work. Verify Add Transaction screen works for all types (expense, income, transfer). These may be working correctly — verify at runtime first.

### Claude's Discretion
- Exact implementation of "Accept All" button for multi-draft (button placement, confirmation dialog)
- Whether category multi-select is needed or single auto-assign with re-select is sufficient
- Additional details toggle implementation (toggle switch vs expand/collapse)
- SnackBar margin adjustment values
- Arabizi detection: prompt-only vs code-level regex assist

</decisions>
