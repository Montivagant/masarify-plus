- Flutter's `SnackBarBehavior.floating` already uses `ScaffoldGeometry` to position above the FAB

- `_adaptiveBottomMargin` in snack_helper.dart already uses `sm` when keyboard is up — now consistent

**No changes needed in snack_helper.dart** — it reads `AppSizes.snackbarBottomMargin` which will now be 8dp. <task type="auto">
  <name>Task 3: Fix stale AI date context across midnight/long sessions</name>
  <files>lib/features/ai_chat/presentation/screens/chat_screen.dart</files>
  <action>
The root cause: `financialContextProvider` captures `DateTime.now()` when its Riverpod dependencies change (wallets, transactions, budgets, etc.), but NOT on each message send. If a user opens the chat at 11pm and sends a message at 1am, the AI still thinks it is the previous day.

**Fix in chat_screen.dart** `_send()` method (around line 92):

Change the financial context read from:
```dart
final financialCtx = ref.read(financialContextProvider);
```
to:
```dart
// Force fresh DateTime.now() capture by invalidating the provider before read.
// Prevents stale date context across midnight or long sessions.
ref.invalidate(financialContextProvider);
final financialCtx = ref.read(financialContextProvider);
```

This is the minimal, correct fix because:
- `ref.invalidate()` disposes the cached provider value, forcing recomputation on next read
- The immediate `ref.read()` triggers fresh computation with a new `DateTime.now()`
- All other provider dependencies (wallets, budgets, etc.) will also be freshly resolved
- No changes needed to the provider definition itself
- Cost is negligible — the provider computation is purely in-memory aggregation
- The provider is only invalidated on user-initiated sends, not on every frame

Do NOT change the provider definition (e.g., by adding a `DateTime` parameter or switching to a family provider) — the invalidate pattern is simpler, localized, and sufficient.
